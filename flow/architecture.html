脆弱点

#1.1SDK 模块顶层捕获 loop ≠ uvicorn loop
根因：lark_oapi 在 import 时抓 loop；uvicorn 用自己的 loop。所有 loop.create_task 必须改 asyncio.get_running_loop().create_task。
#1.2SDK 对 CARD 帧直接 return
根因：_handle_data_frame 只处理 EVENT，CARD 被丢弃 → 所有审批按钮回调丢失（code 200340）。
#1.3挂起 Future 阻止 Ctrl+C
根因：审批 Future 等卡片点击，卡片因其他 bug 没送达 → asyncio 循环无法关闭。
排查入口： 日志有 connected to wss:// 但没 receive pong → loop 错误（#1.1）； 点审批按钮报 200340 → CARD 帧被丢（#1.2）； Ctrl+C 无效 → taskkill /F /PID，查 hook handler 是否有 Future 卡住（#1.3）。
# 典型症状：连得上但收不到消息
INFO: connected to wss://msg-frontier.feishu.cn/ws/v2?...
DEBUG: ping success
# （没有 receive pong，没有 WS frame）

脆弱点

#2.1_dispatch 异常被 asyncio 静默吞
create_task 调度的协程未捕获异常 → 用户无反馈。必须 try/except + send_text 反馈。
#2.2/model 被 /mode startswith 吞
/model 以 /mode 开头，startswith 匹配时短前缀吞长前缀。长前缀必须排前面。
#2.6/mem 用 default_workspace
用户已经 /pwd 切了工作区，但 /mem 仍写到默认路径。所有文件操作都用 session.workspace。
#2.7audit.log 不记录运行中任务
log_command_received 放在 _run_claude 末尾，运行中任务在日志中不可见。开头加 "started"。
#2.8on_card_action 是 sync，不能 await
SDK 要求回调签名 sync。await async 函数 → SyntaxError。拆 cancel_by_user(sync) + cancel_and_wait(async)。
#2.3open_id 与 ALLOWED_USERS 不匹配
open_id 与 app_id 绑定，换应用就变。代码加日志打印实际 open_id。
#2.5pydantic-settings list 字段 + .env 旧字段
list 字段需 JSON 格式（不友好）；重构后旧字段报 extra。改 str + extra=ignore。
排查入口： 发消息无响应 → grep _dispatch 是否有 try/except（#2.1）； /model opus 返回审批帮助 → 命令匹配顺序（#2.2）； 权限被拒 → 实际 open_id vs ALLOWED_USERS（#2.3）； SyntaxError await outside async function → sync 回调里 await 了（#2.8）。

脆弱点

#3.1Windows CreateProcess 不补 .cmd
create_subprocess_exec("claude") 在 Windows 抛 WinError 2。npm shim 是 claude.CMD，必须 shutil.which 解析。
#3.2CLAUDECODE 环境变量触发嵌套检测
父 Claude Code 进程设了 CLAUDECODE，子进程拒绝启动。_build_env 必须 pop。
#3.3Windows 找不到 git-bash
CLI 在 Windows 上必须有 git-bash 才能跑 shell。_build_env 探测候选路径设 CLAUDE_CODE_GIT_BASH_PATH。
#3.4CLI 2.1.71 stdin 不 close 不处理
新版本要求 stdin EOF 才开始处理。send_and_wait 写完必须 writer.close()。失败 subtype 用 startswith("error") 匹配。
#3.5第二条消息 ConnectionResetError
#3.4 副作用：close 后进程退出，但 is_running 看 returncode 还 None。需 _consumed set + _teardown_previous 拆旧 reader。
#3.7subprocess 默认 64KB 单行限制
tool_result 含整文件内容时单行超 64KB → ValueError: Separator not found。limit=10*1024*1024。
#3.9/switch 双重"任务失败"
cancel 触发的 future 用 status=failed 误导用户。改为 cancelled，_run_claude cancelled 静默 return。
#3.10配置隔离后未设 active profile → not logged in
--setting-sources project,local 跳过 ~/.claude/settings.json，凭证只来自 active profile。未 /switch 时 load_active_profile_env() 返回空 dict → 子进程报 not logged in 退出（code=1, cost=$0）。main.py lifespan 启动检查 + 飞书通知 ALLOWED_USERS。
#3.11profile 顶层 model 字段不参与决策
子进程 model 来自 session.last_model → --model arg → env ANTHROPIC_DEFAULT_*_MODEL 映射。profile 的 model 字段只在 /switch 卡片显示用。改 profile 的 model 不会改实际 model，要 /model sonnet。
排查入口： FileNotFoundError: [WinError 2] → #3.1； CLI 1.5s 退出 + text_len=0 → 看 claude stderr: 前缀日志（#3.2 #3.3 #3.4）； 第二条消息 ConnectionResetError → #3.5； Separator/chunk exceed the limit → #3.7； not logged in + cost=$0 time=0s → 未 /switch 选 profile（#3.10）； 改 profile 的 model 不生效 → 用 /model 命令（#3.11）。
架构原则（#3.6）： --input-format stream-json 必须配 -p，每条消息一个进程是固有行为。 不要尝试"去掉 -p 改长驻 stdin"——CLI 不支持。要长驻只能迁移到 Claude Agent SDK。

脆弱点

#4.1CLI 启动失败时空白卡片
错误在 stderr 不在 stdout JSONL。必须 _drain_stderr 后台读取并展示给用户。
#4.2session 与 workspace 绑定
session 在创建时与 cwd 绑定。/pwd 切换 workspace 必须清空 claude_session_id，否则跨目录 resume crash。
#4.3GLM SSE 流无超时
第三方 API 可能不发 [DONE] 或 finish_reason，aiter_lines() 无限等待。必须 read timeout + 异常断开兜底。
#4.4Windows GBK vs UTF-8
默认 GBK 解码 claude 输出会乱码。统一 decode("utf-8", errors="replace")。
排查入口： CLI 无输出 → 看 claude stderr:（#4.1）； 手动验证 → PowerShell echo '你好' | claude -p --output-format stream-json； CLI 自己的 debug 日志 → ~/.claude/debug/； 真实对话历史 → ~/.claude/projects/<encoded-workspace>/<sid>.jsonl。

SDK 响应类构造对比

写法结果
P2CardActionTriggerResponse(toast=t) ❌ TypeError 被 SDK 吞掉 → 飞书收到格式异常 → code 200671

脆弱点

#5.1SDK 响应类不能关键字构造
P2CardActionTriggerResponse/CallBackToast __init__ 不收 kwargs。TypeError 被吞 → 飞书报 200671，但审批逻辑其实成功了。
#5.2send_card 静默吞错
API 返回 code≠0 时只打日志不抛异常，调用方 await future 永远等一个未送达的卡片。必须抛异常 + token 刷新重试。
#5.3内置权限先于 hook
Claude Code 权限层级：内置权限 → PreToolUse hook → 工具执行。permissions 现由 config/claude_settings.json 提供（--settings 加载，含 Bash(*)/Write(*)/Edit(*) 等 allow）。文件缺失或 allow 不全 → Write/Edit 默认拒绝，hook 不触发。
#5.4hook 配置位置（已隔离）
hook 配置在 config/claude_settings.json（openclaw 自有，--settings 显式加载），不写 workspace 的 .claude/。workspace 旧版本注入的 hook 残留由 _strip_openclaw_hooks 按 pre_tool_use.py 脚本路径清理。
排查入口： 点审批按钮报 200671 → 响应类构造方式（#5.1）； 卡片不出现但代码看似正常 → send_card 是否静默失败（#5.2）； claude 说"需要授权"但 hook 没触发 → config/claude_settings.json 缺 permissions.allow 或文件没生成（#5.3）。

Bash 命令风险分类（mode m）

类别示例处理
LOW（白名单）ls, cat, grep, git status, pip list自动放行
HIGH（关键字）rm, git push, pip install, chmod, curl -X POST需审批
HIGH（中文）修改/删除/新建/推送/部署/编辑需审批（正则不需 \b）
未知命令—默认 HIGH（安全优先）
命令链cmd1 && cmd2 / ; / |只检查第一段

脆弱点

#6.1审批通过后 IndexError
ToolCallRecord 只在跳过审批分支创建，通过/拒绝分支漏了。统一在审批后创建 record。
#6.2risk_level str vs enum
router 传 "high"（字符串），manager 调 .value → AttributeError。用 hasattr(risk_level, "value") 兼容。
#6.3run_command 整体 HIGH
ls 和 rm -rf 同走审批 → 体验差。必须命令级分类，未知默认 HIGH。
#6.4DeepSeek 错误静默降级
except 只 logger.warning。返回 error_message 字段，events.py 通知用户。
审批卡片不出现的三步排查：
① config/claude_settings.json 已生成且含 permissions.allow（让内置权限放行，否则 hook 不触发 #5.3）
② send_card 抛异常而非静默（#5.2）
③ hook_log 有记录（确认 hook 脚本被调用）
排查入口： 审批后 IndexError → #6.1； 卡片发送前 AttributeError: 'str' has no .value → #6.2； 只读命令也审批 → #6.3； 服务无法 Ctrl+C → Future 没超时兜底（#1.3）。
